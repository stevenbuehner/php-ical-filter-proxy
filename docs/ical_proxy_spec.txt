PROJEKTSPEZIFIKATION
iCal Proxy & Filter Service (PHP, YAML, sabre/vobject)

1. Ziel des Projekts
Ein leichtgewichtiges PHP-System, das:
- mehrere externe iCal-Feeds lädt
- diese kombiniert (merge)
- anhand von YAML-Regeln filtert und transformiert
- als neue iCal-Feeds öffentlich bereitstellt
- kryptische Abo-Links erzeugt
- Caching auf zwei Ebenen unterstützt
- skalierbar für größere Datenmengen ist

Zentrale Bibliothek:
- sabre/vobject (Parsing & Manipulation von iCalendar-Daten)

5. YAML-Konfiguration
sources:
  gemeinde:
    url: "https://example.com/calendar.ics"
    cache_ttl: "15m"

exports:
  technikdienst:
    title: "Technikdienst"
    slug: "technikdienst"
    token: "random-string"
    cache_ttl: "10m"

    include_sources:
      - source: gemeinde
        filters:
          - type: match
            match:
              summary:
                contains: "Technik"
            on_match: keep

6. Filterlogik
- Regeln werden sequenziell ausgeführt
- `type` definiert den Filtertyp
- `on_match: remove` → entfernt passende Events
- `on_match: keep` → behält passende Events unverändert
- `on_match: transform` → führt `transform[]` aus
- `match.any: true` → diese Regel trifft auf jedes Event zu
- Mehrere Match-Felder innerhalb einer Regel sind per AND verknüpft

7. Match-Operatoren
contains, contains_any, contains_all, not_contains, regex, equals, empty

8. Transformationen (optional)
- Transformationen sind pro Filter konfigurierbar
- jeder Transform ist ein Objekt mit `type`
- Unterstützte Typen: `prefix_text`, `suffix_text`, `replace_text`, `replace_regex`, `remove_property`, `categories_add`, `categories_remove`, `modify_datetime`, `adjust_times`
- Textfelder nutzen `field`
- Kategorien arbeiten über `value`
- Datum/Zeit nutzt `field: start|end`
- `adjust_times` akzeptiert `start` und `end` mit `reference`/`offset`
- Struktur:
  - start.reference: current_start|current_end
  - start.offset: +/-<Zahl>s|m|h
  - end.reference: current_start|current_end
  - end.offset: +/-<Zahl>s|m|h
- All-Day-Events werden ignoriert
- DTEND wird nie vor DTSTART geschrieben; bei Bedarf wird automatisch auf DTSTART korrigiert

15. Fehlerhandling
- defekte Feeds überspringen
- Fehlermeldungen bei defekten Feeds werden geloggt
- Ungültige Regex-Pattern bei replace_regex werden geloggt; betroffene Regel und Transformation werden übersprungen
- Cache-Fallback
- Wenn alle Quellen defekt sind, liefert der Export trotzdem HTTP 200 mit einem gültigen (ggf. leeren) ICS-Feed

Beispiel (Transformation pro Feld im Filter):
transform:
  - type: prefix_text
    field: summary
    value: "[Technik] "
  - type: replace_regex
    field: description
    pattern: "intern"
    replacement: "extern"
